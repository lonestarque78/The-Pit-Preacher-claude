Code Review Assessment

I did another review pass. I agree the truly critical areas look much better, but I would not call the app "clean" yet. What is left is mostly medium-risk hardening and conversion polish.

Findings

1. Medium: some cook subpages still rely too heavily on RLS instead of app-level ownership checks.

apps/web/app/cook/[id]/events/page.tsx loads cooks, cook_events, and cook_logs by cookId only.

apps/web/app/cook/[id]/summary/page.tsx does the same for cook, items, events, logs, outcomes, and tracker notes. Updates/inserts later also only key by cookId or cookLog.id. If RLS is perfect, this is contained; still, the rest of the app has started adding .eq("user_id", user.id), so these should match that pattern.

2. Medium: usage tracking looks wired but not actually used.

incrementUsage is defined but never referenced, and canUseFeature is also unused. The current Preacher limit is enforced separately by counting cook_events, so the daily_usage system is effectively dead code.

Relevant files:
- apps/web/lib/usage/track.ts
- apps/web/lib/gating/canUseFeature.ts

3. Medium: /api/usage likely always returns unauthorized.

apps/web/app/api/usage/route.ts creates a Supabase service client and calls auth.getUser() without passing a bearer token or using the cookie-aware server client. The client hook calls /api/usage?feature=... without an auth header, so this endpoint probably cannot identify the user.

Relevant files:
- apps/web/app/api/usage/route.ts
- apps/web/hooks/useUsage.ts

4. Medium: billing entitlement logic is inconsistent for trialing and past_due.

The webhook can store trialing and past_due, but getTier only accepts status = active. Meanwhile apps/web/lib/billing/isPremium.ts treats trialing as premium. That can make a trialing user premium in one path and free in another.

Relevant files:
- apps/web/app/api/webhook/route.ts
- apps/web/lib/premium.ts
- apps/web/lib/billing/isPremium.ts

5. Low/Medium: checkout accepts arbitrary Stripe price IDs from the client.

apps/web/app/api/billing/create-checkout-session/route.ts only validates that priceId is a non-empty string, then passes it to Stripe. Since the UI sends public env price IDs, the server should whitelist exactly the allowed monthly/annual IDs.

6. Low: old /api/checkout is now duplicate billing plumbing.

It is not referenced by the UI, but it writes directly to subscriptions while the newer flow uses stripe_customers plus webhooks. Remove or clearly deprecate it before launch to avoid future drift.

Verification

npm.cmd exec tsc -- --noEmit passed inside apps/web.

Root npm run check-types failed under PowerShell script policy when called as npm. npm.cmd run check-types exited with no useful Turbo detail, but the app-level type check passed.

Paid Conversion Ideas

The strongest upgrade path is already visible: users get coached during a real cook, then see what premium would have known afterward. Lean into that.

1. Make the post-cook summary the main upsell moment.

After a free cook, show blurred/teased Confidence Score, Fire Control Score, Next Cook Strategy, and What the Preacher learned about your pit. This matches the user's emotional peak: they just finished and want to improve.

2. Add a Cook Replay timeline for paid users.

Competitors and BBQ log apps commonly emphasize time/temp charts, cook logs, photos, and notes. Your version can be more distinctive: "Here is where you lost time. Here is where the bark set. Here is what to do next time."

3. Add photo-based bark/readiness checks as a Backyard/Pitmaster hook.

The code already supports image upload for higher tiers. Make that front-and-center: "Send a photo. Get bark, moisture, wrap, and pull advice." This is visceral and easy to understand.

4. Offer annual plans visibly on /premium.

The billing page has annual price IDs, but the pricing page only shows monthly. Annual plans are an easy revenue lift if positioned as "one season of cooks."

5. Add a Next Cook Strategy Card preview to every completed free cook.

Product-led growth works best when upgrade prompts are tied to value moments and product usage signals, not generic paywalls. The best prompt here is specific: "Your last brisket finished 90 minutes late. Pitmaster can build your next timeline from that history."

Priority

Fix the ownership checks, clean up entitlement consistency, then improve the post-cook upgrade moment. That gives both safety and a better reason for users to pay.
