3. PERFORMANCE OPTIMIZATIONS
============================================================

Two sequential API calls per user message. In apps/web/app/cook/[id]/live/page.tsx
lines 391-413, after receiving the preacher's response, the code immediately
fires a second request to /api/preacher for suggested prompts. Both are
awaited sequentially. The suggested prompts request should be fire-and-forget
(no await) or run in parallel with the insert.

Double isPremium call per gate check. canUseFeature in
apps/web/lib/gating/canUseFeature.ts calls isPremium() directly, then calls
checkUsage() which internally calls isPremium() again — two identical DB
roundtrips.

New Supabase client created on every call. Both incrementUsage and checkUsage
in apps/web/lib/usage/track.ts call createClient(URL, SERVICE_ROLE_KEY) on
every invocation. Same in apps/web/lib/billing/isPremium.ts. Unnecessary
instantiation on every function call.

No pagination on any list query. Cook history, events, and past cooks are all
loaded without limit. A user with 500 cook events pulls all of them into a
single response.

Insights endpoints are force-dynamic but recompute deterministically. The
/api/confidence, /api/fire-control, /api/trends, /api/strategy routes produce
stable output for stable input data. They should use next: { revalidate: 300 }
on their Supabase fetches.

RECOMMENDATION: Parallelize the suggested prompts request, memoize isPremium
per request, add limit() to all list queries, and add revalidation to
analytics endpoints.

4. SECURITY VULNERABILITIES
============================================================

CRITICAL — Race condition in rate limiting
(apps/web/app/api/preacher/route.ts lines 150-202). The pattern reads count,
checks limit, then increments in three separate DB calls. Under concurrent
requests from the same user, both could read request_count: 19, both pass the
>= 20 check, and both increment to 20. Fix with an atomic Postgres function:

  -- Use a Supabase RPC instead of read-then-write
  SELECT increment_rate_limit(user_id, endpoint, window_seconds, max_requests)

/api/billing/portal and /api/billing/create-checkout-session use Bearer token
+ service role key. Both routes do:

  const supabase = createClient(URL, SUPABASE_SERVICE_ROLE_KEY); // full DB access
  const { data: { user } } = await supabase.auth.getUser(accessToken); // client token

Using the service role key to validate a client-provided access token is
an unusual pattern. These routes should use createServerClient() with cookies,
matching the rest of the app.

/api/usage endpoint always returns 401. The route at
apps/web/app/api/usage/route.ts creates a service role client and calls
auth.getUser() with no session — a service role client with no cookies/token
has no user context. This endpoint is completely broken.

env.NEXT_PUBLIC_STRIPE_*_ANNUAL_PRICE_ID! used in getTierFromPriceId. If the
env var is missing, process.env.UNDEFINED_VAR! evaluates to the string
"undefined". The price map then has an entry for the literal key "undefined",
which could match unexpected price IDs and assign the wrong tier.

No request body validation. API routes destructure req.json() with no schema
validation. Malformed or adversarial bodies (e.g., very large imageBase64,
missing required fields) reach business logic unchecked.

RECOMMENDATION: Use an atomic RPC for rate limiting, standardize auth to
cookie-based sessions, validate request bodies with Zod, fix the NEXT_PUBLIC_
env guard for the price map.

different readers/writers.

  apps/web/lib/billing/isPremium.ts
    → reads stripe_subscriptions, checks status

  apps/web/lib/premium.ts

    → writes to subscriptions (via stripe_customer_id)

  apps/web/app/api/stripe/webhook/route.ts
    → writes to stripe_subscriptions (via user_id)

Premium status checks are inconsistent across the codebase. A user's status
in stripe_subscriptions says one thing; subscriptions may say another. There
is no single source of truth.

select('*') on all queries. No projection. Full rows pulled when only one or
two columns are needed.

TOCTOU in pit creation (apps/web/lib/supabase/account.ts lines 82-100).
getPitCount() then createPit() are two separate operations — concurrent
requests can both pass the count < 3 check. This needs a database-level
constraint (e.g., a unique partial index or trigger).

supabase-middleware.ts drops cookie options (apps/web/lib/supabase-middleware.ts
lines 19-20):
  request.cookies.set(name, value)  // options (secure, httpOnly, sameSite) silently dropped

RECOMMENDATION: Consolidate to one subscription table, add select(...) column
projections, enforce pit limit at the DB level, and fix the cookie options
passthrough.

6. API ROUTE DESIGN AND EFFICIENCY
============================================================

/api/checkout/route.ts creates a new Stripe customer on every invocation
(apps/web/app/api/checkout/route.ts lines 35-39):

  const customer = await stripe.customers.create({  // always creates, never checks
    email: user.email,
    metadata: { userId: user.id },
  });

There is no lookup for an existing customer. Every checkout attempt produces a
duplicate cus_xxx object in Stripe. The create-checkout-session route does
this correctly (with an existence check) — the checkout route needs the same
pattern.

Anthropic called via raw fetch instead of the SDK
(apps/web/app/api/preacher/route.ts lines 457-470). The SDK provides typed
responses, automatic retries, and proper error surfacing. The current direct
fetch approach requires manual response parsing and has no retry logic.

Anthropic API version "2023-06-01" is two years old. The model is specified as
claude-sonnet-4-20250514 but the API version is June 2023. Update to
"2024-10-22" or latest stable.

Message type dispatch via string prefix. Six message types are detected with
message.startsWith("OPENING_MESSAGE:") etc. This is fragile — a user who
types "OPENING_MESSAGE:" as their actual question would trigger unintended
behavior. The message type should be a separate field in the request body.

7. AUTHENTICATION AND AUTHORIZATION
============================================================

updateSession from supabase-middleware.ts is never called. The middleware
creates its own Supabase client and checks auth but never calls the session
refresh logic:

  // middleware.ts — creates client, checks getUser(), returns response
  // supabase-middleware.ts — exports updateSession() but middleware.ts never imports it

The Supabase SSR pattern requires the middleware to call updateSession() on
every request to refresh the session token via cookie. Without it, auth tokens
expire and users are logged out unexpectedly. Middleware should be:

  import { updateSession } from '@/lib/supabase-middleware'
  export async function middleware(req: NextRequest) {
    return await updateSession(req)
  }

The Paywall component is client-side only. It renders null during the auth
check and then conditionally shows content. This is not a security boundary —
it's a UX hint. The actual enforcement happens in the API routes (correct),
but the flash of hidden content and reliance on JS for access control is a UX
issue worth addressing.

deletePit in account actions doesn't validate pitId is a UUID before passing
it to Supabase — relies entirely on Supabase's parameterized queries, which is
acceptable but an explicit UUID format check at the boundary is cleaner.

8. STRIPE INTEGRATION AND BILLING LOGIC
============================================================

SUMMARY OF CRITICAL BILLING BUGS:

1. /api/checkout always creates duplicate Stripe customers (missing existence check)

2. Two webhook handlers writing to different tables — only one is active in
   Stripe, so subscription state is permanently out of sync

3. getTierFromPriceId silently maps undefined env vars to the literal string
   "undefined"

4. stripe_subscriptions table (written by stripe/webhook) is checked by
   isPremium in lib/billing/isPremium.ts — but stripe/webhook only writes on
   subscription.created/updated/deleted, never on checkout.session.completed.
   A new subscriber won't appear in stripe_subscriptions unless
   subscription.created fires separately.

5. Stripe API version "2026-03-25.dahlia" in checkout/route.ts is a
   beta/preview version — mixing with the default stable version in other
   routes can produce inconsistent object shapes.

RECOMMENDED REMEDIATION PATH:
1. Delete api/stripe/webhook — consolidate all handling into api/webhook
2. Choose one table (subscriptions is more fully featured with tier) —
   migrate all reads/writes to it
3. Fix the duplicate customer creation in api/checkout
4. Add startup validation that all Stripe env vars are present and not "undefined"

10. TYPESCRIPT USAGE AND TYPE SAFETY
============================================================

typescript: 5.9.2 in apps/web/package.json — as of mid-2025, TypeScript
stable is in the 5.5-5.6 range. 5.9.2 may be a typo for ^5.x or a
pre-release. Verify the intended version.

No Zod schema validation anywhere. All API inputs and FormData values are cast
directly:

An unexpected string passes through as a value that doesn't match the union

  event.data.object as Stripe.Subscription

13. SEO AND METADATA
============================================================

NEXT_PUBLIC_SITE_URL fallback is "" in both apps/web/app/sitemap.ts and
apps/web/app/robots.ts. When this env var is unset, the sitemap produces URLs
like "/" instead of "https://thepitpreacher.com/", and robots.txt points to
"/sitemap.xml" with no domain. Both should fall back to the canonical domain:

  const base = process.env.NEXT_PUBLIC_SITE_URL ?? "https://thepitpreacher.com";

Individual playbook and pitmaster pages likely lack generateMetadata exports.
The global layout.tsx metadata covers the default, but Google will use the
page's own title/description for indexing. These high-value SEO pages (meat
types, pit types, playbook articles) should each export specific metadata.

OG image is /logo.jpeg declared as 1200x630 — verify the actual image is
that resolution; mismatched dimensions cause OpenGraph preview rendering
issues.
